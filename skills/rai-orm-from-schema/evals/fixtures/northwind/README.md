## Schema characteristics

13 base tables: `categories`, `customers`, `customer_demographics`, `customer_customer_demo`, `employees`, `employee_territories`, `order_details`, `orders`, `products`, `region`, `shippers`, `suppliers`, `territories`, `us_states`.

**Why this fixture matters for the SRP:**
- Realistic mid-size schema with mature DDL.
- Composite PK on `order_details` (`order_id`, `product_id`) → external UC + objectified-binary candidate.
- Self-FK on `employees.reports_to → employees.employee_id` → ring-constraint inference target.
- Mild antipattern: `customers` carries denormalized address columns (`address`, `city`, `region`, `postal_code`, `country`).
- `customer_customer_demo` is a pure m:n junction (no extras) — antipattern resolution test.

## Loading the fixture

Postgres:
```bash
bash fetch.sh
createdb northwind
psql -d northwind -f schema.sql
psql -d northwind -f sample_data.sql
```

## Local modifications

None. The split into `schema.sql` and `sample_data.sql` preserves the upstream content verbatim — schema-side statements (DDL + ALTERs) in one file, INSERTs in the other.

## Why fetch-on-demand instead of committing the SQL

The upstream files are MIT-licensed and could be redistributed under the repo's Apache 2.0. We use the fetch-script approach uniformly across both Northwind and TPC-H so the licensing boundary is identical for both — see `notes/phase4-fixture-fetch.md` for the discussion. Running `fetch.sh` reproduces the fixture deterministically from the pinned commit.
