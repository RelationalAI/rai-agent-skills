## Schema characteristics

8 base tables: `NATION`, `REGION`, `PART`, `SUPPLIER`, `PARTSUPP`, `CUSTOMER`, `ORDERS`, `LINEITEM`.

**Why this fixture matters for the SRP:**
- Industry-standard analytics benchmark — clean normalized schema with no antipatterns.
- Composite PK on `LINEITEM` (`L_ORDERKEY`, `L_LINENUMBER`) → external UC.
- Composite PK on `PARTSUPP` (`PS_PARTKEY`, `PS_SUPPKEY`) — also referenced as composite FK from `LINEITEM`.
- No CHECK constraints, no enum-flavored VARCHARs. Tests pure mechanical lift.
- Should produce **zero antipattern flags** — TPC-H is the "clean baseline" benchmark.

## Local modifications

**Important:** the canonical `dss.ddl` from `dbgen` declares only `NOT NULL` constraints; it omits PK and FK declarations. The TPC-H specification (Section 1.4) defines the constraint structure but the dbgen DDL is constraint-free because TPC-H is for benchmarking query performance, not enforcing constraints.

We **append** PK and FK ALTER TABLE statements to `schema.sql` so the SRP's mechanical lift has the full constraint surface to work against. The appended statements are sourced from the TPC-H specification and are bracketed by a comment header:

```sql
-- ----------------------------------------------------------------------------
-- LOCAL MODIFICATION (rai-orm-from-schema, 2026-05-04):
-- The canonical TPC-H dbgen DDL omits PK and FK declarations; the TPC-H
-- specification (Section 1.4) defines them and we add them here so the SRP's
-- mechanical lift has the full constraint surface to work against.
-- ----------------------------------------------------------------------------
```

The PKs and FKs added match TPC-H spec Section 1.4 exactly — no inference or extension. Per the TPC-H specification:
- PK on every table.
- FK from `NATION → REGION`.
- FK from `SUPPLIER → NATION`, `CUSTOMER → NATION`.
- FK from `PARTSUPP → PART`, `PARTSUPP → SUPPLIER`.
- FK from `ORDERS → CUSTOMER`.
- FK from `LINEITEM → ORDERS`, FK from `LINEITEM → PARTSUPP` (composite).

This augmentation is documented inline in `schema.sql` so anyone reading it sees what we changed and why.
