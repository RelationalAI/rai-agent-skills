# ORM → PyRel Translation

The four-tier translation rules from `model.orm.yaml` to `model.py`. Used at SRP **Step 10**.

Load at SRP Step 10. The full output is `model.py`; this reference walks the rules and shows canonical patterns per construct.

Cross-references:
- YAML input shape → [representation-format.md](representation-format.md)
- Per-constraint vocabulary (which tier each lands in) → [constraint-reference.md](constraint-reference.md)
- PyRel idioms (canonical syntax) → `rai-pyrel-coding/SKILL.md`

## Halpin-copyright discipline

Translation rules are paraphrased from Halpin's published work; examples are ours. Concept names, property names, fact-type readings used in this reference are illustrative — not copied from any published source.

## Four tiers

Every YAML entry lands in exactly one tier:

| Tier | What it covers | PyRel construct |
|---|---|---|
| **Mechanical** | Object types; uniqueness-on-role binary fact types; multi-valued binary; NOT NULL; subtypes; value enumerations; decimal types | `Concept(identify_by=...)`, `Property` (FD-enforced), `Relationship`, `model.Enum`, `Number.size(p,s)`, `extends=[Parent]` + `define...where` |
| **Heuristic** | Objectified binary; subtype recovery from TYPE columns; same-type fact types; unaries | Junction concept with `identify_by={"a": A, "b": B}`; named refs `{Stock:stock1}`; unary `Relationship` |
| **`model.require()` (verbose)** | Ring constraints; counted frequency; external uniqueness over joins; subset / exclusion / equality; value-comparison; cardinality | `model.require(...)` with explanatory comments |
| **Truly not translated** | Deontic constraints; FORML 2 textual constraints; (acyclic if PyRel lacks transitive closure) | `# DEONTIC NOTE:` / `# NOTE:` comments; constraint preserved in YAML for documentation |

The tier is determined per YAML entry, not per constraint type. A `value` constraint with enum lands in mechanical (`model.Enum`); the same `value` constraint with a numeric range lands in `model.require`. The translator inspects each entry's full shape.

## Output structure

A complete `model.py` has the following structure:

```python
"""
Generated by rai-orm-from-schema on 2026-05-04.

For production use, set `implicit_properties: false` in raiconfig.yaml
to catch typos at definition time.
"""

# ---- Imports ----
from relationalai.semantics import (
    Model, Float, Integer, String, Date, DateTime, Boolean, Number,
    count, sum, min, max, avg, per, distinct,
)

# ---- Model ----
model = Model("orders")

# ---- Concepts (object types) ----
Customer = model.Concept("Customer", identify_by={"id": Integer})
Order = model.Concept("Order", identify_by={"id": Integer})
# ...

# ---- Properties / Relationships (binary fact types) ----
Customer.email = model.Property(f"{Customer} has email {String:email}")
Order.customer = model.Property(f"{Order} placed by {Customer:customer}")
# ...

# ---- Subtypes ----
PhysicalProduct = model.Concept("PhysicalProduct", extends=[Product])
model.define(PhysicalProduct(Product)).where(Product.type == "PHYSICAL")
# ...

# ---- Enums ----
OrderStatus = model.Enum("OrderStatus", ["PENDING", "PAID", "SHIPPED", "DELIVERED", "CANCELLED"])

# ---- Verbose-tier constraints ----
# Each Customer.age must be non-negative.
model.require(Customer.age >= 0)

# Parenthood is acyclic.
model.require(...)  # see Acyclic note below

# ---- Deontic notes ----
# DEONTIC NOTE: each Customer should have at least one Order.

# ---- Textual / NOTE comments ----
# NOTE (textual constraint, source: user-supplied):
# Each Employee who has Rank 'NonExec' uses at most one CompanyCar.

# ---- Review flags (from One-shot mode) ----
# REVIEW PROPOSAL: the following constraint was proposed by LLM inference and
# has not been confirmed by a user.
model.require(Order.delivered_at <= Order.shipped_at + Duration("30 days"))
```

The exact section ordering above is the canonical layout. Phase 4 evals check that the output adheres.

## Tier 1 — Mechanical

### Entity types

```yaml
object_types:
  - id: customer
    name: Customer
    kind: entity
    reference:
      mode: popular
      value_type: customer_id
```

with the value type:

```yaml
  - id: customer_id
    name: CustomerId
    kind: value
    primitive: Integer
```

Translates to:

```python
Customer = model.Concept("Customer", identify_by={"id": Integer})
```

**Rules:**
- `identify_by` field name = the role name on the identity fact type, default = `id` for popular reference with primitive value type. For non-default, use the role name from the YAML.
- The primitive on the right of `identify_by` matches the `value_type`'s primitive (or its name if it's a non-primitive value type — but most popular references use primitives).
- **Do NOT also declare a `Property` for the identity field.** `identify_by` auto-creates the property; double-declaration creates duplicates.

### Composite-identifier entities (external reference)

```yaml
object_types:
  - id: order_item
    name: OrderItem
    kind: entity
    reference:
      mode: external

# corresponding external UC in top-level constraints:
constraints:
  - type: uniqueness
    scope: external
    role_refs:
      - { fact_type: order_item_for_order, role: order }
      - { fact_type: order_item_for_product, role: product }
    preferred: true
```

Translates to:

```python
OrderItem = model.Concept("OrderItem", identify_by={"order": Order, "product": Product})
```

**Rule:** the `identify_by` keys are the role names from the external UC's `role_refs`. The values are the referenced object types' Python names (concept references, not strings).

### Value types

Primitive value types are not declared as concepts in PyRel — they're used directly in property f-strings. The YAML's value-type entries are reference shapes, not output.

```yaml
object_types:
  - id: email_address
    name: EmailAddress
    kind: value
    primitive: String
```

In PyRel, `String` is used directly:

```python
Customer.email = model.Property(f"{Customer} has email {String:email}")
```

Custom value types extending a primitive get a Concept declaration:

```yaml
  - id: email
    name: Email
    kind: value
    primitive: String
    extends_primitive: true                  # rare; opt-in
```

```python
Email = model.Concept("Email", extends=[String])
```

The `extends_primitive: true` flag is a hint for the translator — most value types don't need their own concept declaration.

### Binary fact types — Property (uniqueness on one role)

```yaml
fact_types:
  - id: customer_has_email
    reading: "{Customer} has email {EmailAddress}"
    roles:
      - { object_type: customer, role_name: customer }
      - { object_type: email_address, role_name: email }
    constraints:
      - type: uniqueness
        roles: [customer]
        source: explicit
        status: confirmed
```

Translates to:

```python
Customer.email = model.Property(f"{Customer} has email {String:email}")
```

**Rules:**
- `Property` is single-valued by definition. The uniqueness on the `customer` role becomes the FD enforcement.
- The f-string preserves the YAML's reading verbatim, with `{Type:role_name}` replacing each YAML role.
- The property name (left side, `Customer.email`) = the role name on the value side.
- For value types that are primitives, use the primitive name (`String`, `Integer`) in the f-string.

### Binary fact types — Property where value type is another entity (FK fact type)

```yaml
fact_types:
  - id: order_placed_by_customer
    reading: "{Order} placed by {Customer}"
    roles:
      - { object_type: order, role_name: order }
      - { object_type: customer, role_name: customer }
    constraints:
      - type: uniqueness
        roles: [order]
```

```python
Order.customer = model.Property(f"{Order} placed by {Customer:customer}")
```

**Rules:**
- The named ref `{Customer:customer}` distinguishes the role within the predicate.
- Property name = role name on the value side.

### Binary fact types — Relationship (m:n; spanning UC)

```yaml
fact_types:
  - id: order_item_tagged_with
    reading: "{OrderItem} tagged with {Tag}"
    roles:
      - { object_type: order_item, role_name: orderItem }
      - { object_type: tag, role_name: tag }
    constraints:
      - type: uniqueness
        roles: [orderItem, tag]
```

```python
OrderItemTagging = model.Relationship(f"{OrderItem} tagged with {Tag}")
```

**Rules:**
- m:n binary fact types translate to `Relationship`, not `Property`.
- Relationship name = camelCase concatenation of the two object types + a verb noun, or use the YAML's id when no clean name emerges (`order_item_tagging`).

### Subtypes

```yaml
object_types:
  - id: physical_product
    name: PhysicalProduct
    kind: entity
    subtype_of: [product]
    inherits_reference: true
    derivation: derived
    derivation_rule: "Each PhysicalProduct is a Product whose Type is 'PHYSICAL'."
```

```python
PhysicalProduct = model.Concept("PhysicalProduct", extends=[Product])
model.define(PhysicalProduct(Product)).where(Product.type == "PHYSICAL")
```

**Rules:**
- `extends=[Parent]` is mandatory for subtypes.
- Membership rule comes from `derivation_rule` translated into a `define...where` predicate.
- For multi-supertype subtypes (`subtype_of: [a, b]`), use `extends=[A, B]` and the membership rule needs a conjunction.
- For `derivation: asserted` subtypes (no rule, instances assigned by data), omit the `define...where`; membership is implied by the data binding.

### Value constraints — enumeration

```yaml
constraints:
  - type: value
    object_type: order_status
    allowed: ['PENDING', 'PAID', 'SHIPPED', 'DELIVERED', 'CANCELLED']
```

```python
OrderStatus = model.Enum("OrderStatus", ["PENDING", "PAID", "SHIPPED", "DELIVERED", "CANCELLED"])
```

The class-based form is also valid:

```python
class OrderStatus(model.Enum):
    PENDING = "PENDING"
    PAID = "PAID"
    SHIPPED = "SHIPPED"
    DELIVERED = "DELIVERED"
    CANCELLED = "CANCELLED"
```

The translator emits the functional form by default; class form is a v1.5 stretch when string-typed values aren't sufficient.

### NOT NULL → Property (mandatory single-valued)

PyRel `Property` is always single-valued. NOT NULL on the FK side of a 1:n binary fact type is implicit in `Property` semantics — no extra translation needed.

For NOT NULL on the value side of a m:n `Relationship`, see the `model.require` tier below.

### Decimal types

```yaml
object_types:
  - id: price
    name: Price
    kind: value
    primitive: Number
    precision: 10
    scale: 2
```

In PyRel f-strings:

```python
Order.total = model.Property(f"{Order} has total {Number.size(10, 2):total}")
```

**Rules:**
- Always use `Number.size(precision, scale)` for non-integer decimals.
- Bare `Number` causes inference issues — never emit it.
- Scale-0 decimals translate to `Integer`, not `Number.size(p, 0)` (per `rai-pyrel-coding`).

### Binary primitive

```yaml
object_types:
  - id: binary_blob
    name: BinaryBlob
    kind: value
    primitive: Binary
```

PyRel translation: emit a value type using `Binary` directly. In f-strings, `{Binary:fieldName}` works the same way as any other primitive:

```python
from relationalai.semantics import Binary

# ...
Category.picture = model.Property(f"{Category} has picture {Binary:picture}")
Employee.photo = model.Property(f"{Employee} has photo {Binary:photo}")
```

**Rules:**
- Use `Binary` for any column with a binary/blob storage type (Postgres `bytea`, MySQL `BLOB`, etc.).
- Don't try to translate the actual contents — PyRel treats Binary as an opaque value type.
- If the user wants typed binary (e.g., images vs PDFs), emit a separate value constraint at the user-supplied tier (Step 9d).

## Tier 2 — Heuristic

### Objectified binary

```yaml
fact_types:
  - id: student_enrolled_in_course
    reading: "{Student} enrolled in {Course}"
    roles:
      - { object_type: student, role_name: student }
      - { object_type: course, role_name: course }
    objectified_as: enrolment
    objectification_independent: true

object_types:
  - id: enrolment
    name: Enrolment
    kind: entity
    reference: { mode: external }
```

```python
Enrolment = model.Concept("Enrolment", identify_by={"student": Student, "course": Course})

# Attributes of the objectification translate as Properties on Enrolment:
Enrolment.grade = model.Property(f"{Enrolment} resulted in grade {String:grade}")
```

**Rules:**
- The objectification entity uses composite `identify_by` keyed on the role names of the underlying binary.
- This is the canonical pattern for junction-with-extras (per `rai-pyrel-coding` line 159).

### Subtype recovery from TYPE columns

This is upstream from translation — Step 7 antipattern detection emits the subtype YAML during SRP. Step 10 just translates the resulting subtype entries via the mechanical rules above.

### Same-type fact types

```yaml
fact_types:
  - id: stock_pair_covar
    reading: "{Stock} and {Stock} have covar {Float}"
    roles:
      - { object_type: stock, role_name: stock1 }
      - { object_type: stock, role_name: stock2 }
      - { object_type: float, role_name: covar }
```

```python
Stock.covar = model.Property(f"{Stock:stock1} and {Stock:stock2} have {Float:covar}")
```

**Rules:**
- Named refs (`{Stock:stock1}`, `{Stock:stock2}`) disambiguate the same type appearing twice.
- Without named refs, PyRel can't tell which role is which.

### Unary fact types

```yaml
fact_types:
  - id: order_is_rush
    reading: "{Order} is rush priority"
    roles:
      - { object_type: order, role_name: order }
```

```python
Order.is_rush = model.Property(f"{Order} is rush priority")
# OR for a true unary relationship without value:
order_is_rush = model.Relationship(f"{Order} is rush priority")
```

**Rules:**
- For Boolean-flag-style unaries (most common), translate as `Property` on the entity.
- For pure unary relationships (rare), translate as `Relationship`.

## Tier 3 — `model.require()` (verbose-but-expressible)

Every constraint in this tier translates to one or more `model.require(...)` calls.

### Ring constraints

```yaml
constraints:
  - type: ring
    variant: irreflexive
    fact_type: person_parent_of_person
```

```python
# parenthood is irreflexive
model.require(model.not_(Person.parent_of(Person, parent=Person, child=Person)))
```

For `acyclic`:

```python
# parenthood is acyclic
# NOT-TRANSLATED-IF-NO-TRANSITIVE-CLOSURE:
# Acyclic requires transitive-closure expression, which depends on PyRel's
# support for recursive rule definitions. If supported, emit using a defined
# transitive-closure rule and require its irreflexivity. If not supported,
# emit only as # NOTE: comment and leave constraint in YAML for documentation.
```

The exact `acyclic` translation depends on PyRel version. Phase 3 should verify; v0.1 emits a NOTE comment if transitive closure is unsupported.

For `asymmetric`:

```python
# parenthood is asymmetric
model.require(Person.parent_of(parent=Person, child=Person)
              .implies(model.not_(Person.parent_of(parent=child, child=parent))))
```

For `transitive`:

```python
# parenthood is transitive (this is unusual for parent-of; typically used for "ancestor of")
model.require((Person.parent_of(parent=A, child=B) & Person.parent_of(parent=B, child=C))
              .implies(Person.parent_of(parent=A, child=C)))
```

Each ring variant is a separate `model.require` call (per the multi-variant policy).

### Frequency (counted)

```yaml
constraints:
  - type: frequency
    scope: internal
    roles: [member]
    bound: { min: 12, max: 12 }
    fact_type: jury_has_member
```

```python
# each jury has exactly 12 members
model.require(count(Member.of(jury)).per(jury) == 12)
```

For ranges:

```python
# each panel has 4-7 experts
model.require((count(Expert.on(panel)).per(panel) >= 4)
              & (count(Expert.on(panel)).per(panel) <= 7))
```

### External uniqueness over joins

```yaml
constraints:
  - type: uniqueness
    scope: external
    role_refs:
      - { fact_type: state_in_country, role: state }
      - { fact_type: state_has_code, role: code }
    preferred: false
```

When the external UC corresponds to a junction-shaped table → mechanical (covered above as composite-`identify_by`).

When the external UC is over true binary fact types and the entity isn't a junction → `model.require`:

```python
# (state, code) combination uniquely identifies a State
model.require(count(State.country_state_code_pair).per(country, code) == 1)
```

### Subset

```yaml
constraints:
  - type: subset
    subset_role_seq: [{ fact_type: person_smokes, role: person }]
    superset_role_seq: [{ fact_type: person_is_cancer_prone, role: person }]
```

```python
# every smoker is cancer-prone
model.require(Person.smokes.implies(Person.cancer_prone))
```

For multi-role subset (join subset):

```python
# if an advisor serves in a country, they speak a language used in that country
model.require(...)  # join-style implication; complex
```

Use `model.where(...)` for join paths in subset/equality/exclusion. See `rai-pyrel-coding` for the syntax.

### Equality

```yaml
constraints:
  - type: equality
    role_seq_a: [{ fact_type: patient_has_systolic_bp, role: patient }]
    role_seq_b: [{ fact_type: patient_has_diastolic_bp, role: patient }]
```

```python
# A patient has systolic-BP iff they have diastolic-BP
model.require(Patient.systolic_bp.implies(Patient.diastolic_bp))
model.require(Patient.diastolic_bp.implies(Patient.systolic_bp))
```

Or:

```python
model.require(Patient.systolic_bp == Patient.diastolic_bp)  # presence-equality
```

### Exclusion

```yaml
constraints:
  - type: exclusion
    role_seq_a: [{ fact_type: person_is_married, role: person }]
    role_seq_b: [{ fact_type: person_is_widowed, role: person }]
```

```python
# no person is both married and widowed
model.require(model.not_(Person.is_married & Person.is_widowed))
```

### Exclusive-or

```yaml
constraints:
  - type: exclusive-or
    role_seqs:
      - [{ fact_type: academic_is_male, role: academic }]
      - [{ fact_type: academic_is_female, role: academic }]
```

```python
# Each Academic is male xor female
model.require((Academic.is_male & model.not_(Academic.is_female))
              | (Academic.is_female & model.not_(Academic.is_male)))
```

### Value range

```yaml
constraints:
  - type: value
    object_type: age
    range: { min: 0, min_inclusive: true }
```

```python
model.require(Person.age >= 0)
```

For closed range:

```python
model.require((Account.discount_pct >= 0) & (Account.discount_pct <= 100))
```

For multi-range:

```python
model.require(((LatitudeOffset.value >= -100) & (LatitudeOffset.value <= -20))
              | ((LatitudeOffset.value >= 40) & (LatitudeOffset.value <= 100)))
```

### Value-comparison

```yaml
constraints:
  - type: value-comparison
    operator: ">="
    role_a: { fact_type: project_started, role: date }
    role_b: { fact_type: project_ended, role: date }
```

```python
model.require(Project.end_date >= Project.start_date)
```

Operator mapping (including the v0.1 extension `!=`):

| YAML `operator` | PyRel translation |
|---|---|
| `<` | `<` |
| `<=` | `<=` |
| `=` | `==` |
| `>=` | `>=` |
| `>` | `>` |
| `!=` (v0.1 extension; "must be distinct" case) | `!=` (or equivalently `model.not_(a == b)`) |

Example for `!=` — the InventoryTransfer from/to constraint:

```yaml
constraints:
  - type: value-comparison
    operator: "!="
    role_a: { fact_type: inventory_transfer_from, role: fromWarehouse }
    role_b: { fact_type: inventory_transfer_to, role: toWarehouse }
```

```python
# Transfers between two warehouses must have distinct from/to.
model.require(InventoryTransfer.from_warehouse != InventoryTransfer.to_warehouse)
```

### Role-scope value constraints (hybrid `object_type` + `role_ref`)

When a value constraint targets a specific role on one fact type — rather than every appearance of a value type — the YAML uses the hybrid form: `object_type` names the primitive being constrained, and `role_ref` narrows the scope.

```yaml
constraints:
  - type: value
    object_type: integer            # the primitive being constrained
    role_ref: { fact_type: inventory_quantity_on_hand, role: quantityOnHand }
    range: { min: 0, min_inclusive: true }
```

```python
# Narrow to the specific role; doesn't constrain every Integer.
model.require(Inventory.quantity_on_hand >= 0)
```

PyRel translation rule: walk to the `Concept.property` syntax matching the named fact type + role, then emit the comparison or range check. This is `model.require()` tier — no mechanical shortcut.

### Cardinality (object)

```yaml
constraints:
  - type: cardinality
    scope: object
    object_type: president
    bound: { min: 1, max: 1 }
```

```python
# exactly one President
model.require(count(President) == 1)
```

### Cardinality (role)

```yaml
constraints:
  - type: cardinality
    scope: role
    role_ref: { fact_type: politician_is_president, role: politician }
    bound: { max: 1 }
```

```python
model.require(count(Politician.is_president) <= 1)
```

### Subtype-partition (top-level constraint side)

The subtype membership is mechanical (covered above). The exclusivity / exhaustiveness sides land here:

```yaml
constraints:
  - type: subtype-partition
    supertype: product
    subtypes: [physical_product, digital_product, subscription_product]
    exclusive: true
    exhaustive: true
```

Exclusivity:

```python
# no Product is more than one subtype
model.require(model.not_(PhysicalProduct & DigitalProduct))
model.require(model.not_(PhysicalProduct & SubscriptionProduct))
model.require(model.not_(DigitalProduct & SubscriptionProduct))
```

Exhaustiveness:

```python
# every Product is at least one subtype
model.require(Product.implies(PhysicalProduct | DigitalProduct | SubscriptionProduct))
```

## Tier 4 — Truly not translated

### Deontic constraints

Any constraint with `modality: deontic` translates to a comment, regardless of tier:

```python
# DEONTIC NOTE: each Customer should have at least one Order.
# (deontic — should rule, possibly violated; not enforced by model.require)
```

PyRel's `model.require` is alethic-only. Deontic constraints must NOT be translated to `model.require` — that would over-enforce.

### Textual / FORML 2 constraints

```yaml
constraints:
  - type: textual
    expression: "Each Employee who has Rank 'NonExec' uses at most one CompanyCar."
    formal_language: natural
```

```python
# NOTE (textual constraint, source: user-supplied):
# Each Employee who has Rank 'NonExec' uses at most one CompanyCar.
```

v0.1 doesn't parse FORML 2. v1.5+ may add a parser; until then, textual constraints emit as documentation comments.

### Acyclic without transitive closure

If PyRel's version doesn't support recursive rule definitions for transitive closure, `acyclic` ring constraints fall to this tier:

```python
# NOT TRANSLATED: parenthood is acyclic.
# PyRel does not support transitive-closure expression in this version.
# Constraint is documented in the YAML for future enforcement.
```

Phase 3 verifies PyRel's transitive-closure support and updates this section accordingly.

## Boolean-operator discipline

PyRel does **not** use Python's `and`, `or`, `not`. Always use the PyRel operators:

| Wrong (Python) | Right (PyRel) |
|---|---|
| `(x >= 1) and (y >= 1)` | `(x >= 1) & (y >= 1)` |
| `(x >= 1) or (y >= 1)` | `(x >= 1) \| (y >= 1)` |
| `not x_assigned` | `model.not_(x_assigned)` |
| `if condition else fallback` | `value \| fallback` (ordered fallback) |

Using Python operators triggers `[Invalid operator] Cannot use python's 'bool check'` at compile time. The translator must always emit PyRel-correct operators in `model.require` expressions.

## Module-level definitions

PyRel models are declared at module level, not inside functions. The translator emits all declarations at the top of `model.py`:

```python
# Good — module-level declarations
Customer = model.Concept("Customer", identify_by={"id": Integer})

def query_orders(customer):
    return Customer.orders  # uses the module-level Customer
```

```python
# Wrong — declarations inside functions
def build_model():
    Customer = model.Concept("Customer", ...)  # module introspection won't see this
```

Reasoner-specific code (solvers, queries) goes in functions; the model itself stays at module level.

## Imports

Standard imports for any generated `model.py`:

```python
from relationalai.semantics import (
    Model, Float, Integer, String, Date, DateTime, Boolean,
    count, sum, min, max, avg, per, distinct,
)
```

Add as needed:

```python
from relationalai.semantics import Number  # for decimal types with scale > 0
```

The translator emits a minimal import list — only types and aggregates the model actually uses.

## Header / strict-mode comment

Every generated `model.py` opens with the header:

```python
"""
Generated by rai-orm-from-schema on <ISO timestamp>.

For production use, set `implicit_properties: false` in raiconfig.yaml
to catch typos at definition time. Without it, accessing an undeclared
property silently creates a new one — masking schema-recovery errors.
"""
```

The strict-mode recommendation is a comment, not enforced — Phase 4 doesn't validate raiconfig.yaml. But the comment is the canonical reminder per `rai-pyrel-coding`.

## Review-flag comments (One-shot mode)

When the user runs One-shot mode, some proposals stay `proposed` in the YAML. Their PyRel translation is emitted with a flag comment:

```python
# REVIEW PROPOSAL (source: llm-inferred, status: proposed):
# Returns must reference orders that have been delivered.
model.require(Return.order.implies(Order.status == "DELIVERED"))
```

```python
# REVIEW MODALITY (source: common-sense, status: confirmed, modality default = alethic):
# Parenthood is acyclic. (User did not label modality at Step 9e.)
model.require(...)
```

These flags help the user spot what to review when they edit `model.py` post-generation.

## What this translation does NOT cover in v0.1

- **PyRel solver-context code.** The translator only emits the model declarations. Solver invocation, problem setup, optimisation results — out of scope; user adds these manually for their use case.
- **PyRel queries / `model.where()` for non-constraint use.** The translator emits constraints; query construction is the user's job.
- **PyRel data binding.** `model.Table(...)` for Snowflake or `model.data(...)` for prototyping is not generated — user adds in their own code.
- **Multi-model composition.** v0.1 emits one `Model(...)` per YAML; cross-model references are out of scope.

These gaps are by design — the translation is conceptual-model-only. Application-specific code is the user's job, post-generation.
